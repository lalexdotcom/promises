# Plan 1: Fix Runtime Bugs (BUG-01, BUG-02)

## Objective

Fix two runtime bugs that silently break core functionality: a wrong boolean guard that prevents per-promise timeouts from ever firing, and a `timeout()` utility that incorrectly rejects its outer promise when the inner promise resolves late, while also failing to forward inner rejections.

## Context

**BUG-01** — In `runNext()`, the condition `Number.isNaN(timeout) && timeout > 0` is logically impossible: `Number.isNaN(x)` is `true` only when `x` is `NaN`, and `NaN > 0` is always `false`. The intent is the inverse: apply a timeout when `timeout` *is* a valid positive number. This means `enqueue(fn, 5000)` never wraps `fn` in a timeout — the per-promise timeout feature is entirely broken.

**BUG-02** — In `timeout()` in `src/utils.ts`, when the inner promise `p` resolves *after* the timeout has already fired (`isTooLate === true`), the code calls `rej()` with no argument. This corrupts the outer promise (which already rejected with `TimeoutError`) by triggering the reject handler a second time (if not yet settled) or producing an unhandled rejection. The fix is to silently ignore late resolutions. Additionally, rejections from `p` are not forwarded at all — `.catch()` is never attached, so if `p` rejects it becomes an unhandled promise rejection.

---

## Files to Modify

- `src/pool.ts` — fix the boolean guard in `runNext()`
- `src/utils.ts` — fix late-resolution behaviour and add rejection forwarding in `timeout()`

## Implementation

### Task 1 — Fix the `runNext()` guard (BUG-01)

**File:** `src/pool.ts`

**Current code:**
```ts
					const nextPromise =
						Number.isNaN(timeout) && timeout > 0 ? timeoutPromise(generator(), timeout) : generator();
```

**Change to:**
```ts
					const nextPromise =
						!Number.isNaN(timeout) && timeout > 0 ? timeoutPromise(generator(), timeout) : generator();
```

**Reason:** `Number.isNaN(timeout)` is `true` only for `NaN`; combining it with `&& timeout > 0` makes the whole expression permanently `false`. Negating `Number.isNaN` gives the correct semantics: "timeout is a real, positive number → wrap in timeout".

---

### Task 2 — Fix late-resolution and propagate rejections (BUG-02)

**File:** `src/utils.ts`

**Current code:**
```ts
export function timeout<T>(p: Promise<T>, delay: number): Promise<T> {
	return new Promise((res, rej) => {
		let isTooLate = false;
		let isResolved = false;
		const to = setTimeout(() => {
			if (!isResolved) {
				isTooLate = true;
				clearTimeout(to);
				rej(new TimeoutError('Promise timed out'));
			}
		}, delay);
		p.then((v) => {
			if (isTooLate) rej();
			else {
				isResolved = true;
				clearTimeout(to);
				res(v);
			}
		});
	});
}
```

**Change to:**
```ts
export function timeout<T>(p: Promise<T>, delay: number): Promise<T> {
	return new Promise((res, rej) => {
		let isTooLate = false;
		let isResolved = false;
		const to = setTimeout(() => {
			if (!isResolved) {
				isTooLate = true;
				clearTimeout(to);
				rej(new TimeoutError('Promise timed out'));
			}
		}, delay);
		p.then((v) => {
			if (!isTooLate) {
				isResolved = true;
				clearTimeout(to);
				res(v);
			}
			// late resolution is silently ignored — the outer promise already rejected
		}).catch((err) => {
			if (!isTooLate) {
				isResolved = true;
				clearTimeout(to);
				rej(err);
			}
		});
	});
}
```

**Reason:** Two independent fixes:

1. **Late resolution**: replace `if (isTooLate) rej()` with `if (!isTooLate) { ... }` — when the timeout has already fired, silently do nothing rather than calling `rej()` again.
2. **Rejection forwarding**: attach `.catch()` on `p`. Without this, if `p` rejects, the outer promise never rejects and the browser/Node runtime emits an unhandled rejection warning.

## Notes

- The `enqueue` signature defaults `timeout` to `Number.NaN` when none is supplied, so the guard `!Number.isNaN(timeout) && timeout > 0` correctly skips the timeout wrapper for the common case.
- The `isResolved` flag in `timeout()` guards against the timer firing after `p` has already resolved; the `isTooLate` flag guards in the other direction. Both remain necessary after this fix.
- No new dependencies are introduced; both fixes are pure logic changes.
