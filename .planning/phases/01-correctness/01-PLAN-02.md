# Plan 2: Dead Code Removal and Type Improvements (BUG-03, BUG-04, TYPES-01, TYPES-02, TYPES-03)

## Objective

Remove dead and confusing code (`pending` duplicate getter, commented-out `#emit` calls) and improve the public TypeScript types so that `pool.parallel()`, `pool.serial()`, and the `PromisePool` interface all use precise, `any`-free signatures that let callers get full type inference on resolved values.

## Context

**BUG-03** — `PromisePoolImpl` has a `pending` getter that is byte-for-byte identical to the `waiting` getter (`return this.#enqueued.length`). `pending` does not appear in the `PromisePool` interface. Having both creates confusion about which is correct to use. Resolution: remove `pending`.

**BUG-04** — Two `// this.#emit('next')` calls are commented out inside `promiseDone` and `promiseRejected`. Commented-out code with no explanation creates confusion about whether these are intentional omissions or forgotten removals. Resolution: delete both comment lines.

**TYPES-01 / TYPES-02** — `pool.parallel()` and `pool.serial()` are typed as `(commands: PromiseFunction[], ...) => Promise<any[]>`. Callers get no type information from the returned promise. The fix is a generic signature using a mapped type spread so TypeScript infers:
- Homogeneous arrays → `Promise<T[]>` (e.g. `(() => Promise<string>)[]` → `Promise<string[]>`)
- Heterogeneous tuples → `Promise<[R1, R2, ...]>` (e.g. `[() => Promise<string>, () => Promise<number>]` → `Promise<[string, number]>`)

**TYPES-03** — Private fields in `PromisePoolImpl` use `any` (`#running: Promise<any>[]`, `#promise: Promise<any[]>`, `result: any[]`, `#resolve`, `#reject`). These should use `unknown` to stay consistent with the `PromisePool` interface which already declares `promise: Promise<unknown[]>`.

### Task 3 — Replace `any` with `unknown` in private fields (TYPES-03)

**File:** `src/pool.ts`

**Current code (private field declarations, inside `PromisePoolImpl`):**
```ts
	#running: Promise<any>[] = [];
	#enqueued: QueuedPromise[] = [];
	private result: any[] = [];

	#isStarted = false;
	#isClosed = false;
	#isResolved = false;

	#promise: Promise<any[]>;
	#resolve!: (...args: any[]) => void;
	#reject!: (...args: any[]) => void;
```

**Change to:**
```ts
	#running: Promise<unknown>[] = [];
	#enqueued: QueuedPromise[] = [];
	private result: unknown[] = [];

	#isStarted = false;
	#isClosed = false;
	#isResolved = false;

	#promise: Promise<unknown[]>;
	#resolve!: (value: unknown[]) => void;
	#reject!: (reason?: unknown) => void;
```

**Reason:** The public `PromisePool` interface already declares `promise: Promise<unknown[]>`. Using `any` internally undermines the type contract. `unknown` preserves the soundness guarantee: callers must narrow before using values, which is the correct behaviour for a generic pool that doesn't know the resolved type of arbitrary promises.

### Task 4 — Typed return for `pool.parallel()` (TYPES-01)

**File:** `src/pool.ts`

**Current code:**
```ts
		parallel: (commands: PromiseFunction[], options?: PoolOptions): Promise<any[]> => {
			if (!commands.length) return Promise.resolve([]);
			const parallelPool = new PromisePoolImpl({ concurrency: Number.POSITIVE_INFINITY, ...options });
			for (const cmd of commands) parallelPool.enqueue(cmd);
			return parallelPool.close();
		},
```

**Change to:**
```ts
		parallel: <T extends PromiseFunction[]>(
			commands: [...T],
			options?: PoolOptions,
		): Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }> => {
			if (!commands.length) return Promise.resolve([]) as Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>;
			const parallelPool = new PromisePoolImpl({ concurrency: Number.POSITIVE_INFINITY, ...options });
			for (const cmd of commands) parallelPool.enqueue(cmd);
			return parallelPool.close() as Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>;
		},
```

**Reason:** The `[...T]` spread causes TypeScript to infer `T` as a tuple when a literal array is passed, and as an array type when a variable array is passed. The mapped type `{ [K in keyof T]: Awaited<ReturnType<T[K]>> }` then naturally resolves to a typed tuple or `U[]` respectively — matching the behaviour of `Promise.all`. The two `as` casts are needed because the implementation uses an untyped internal pool whose `close()` returns `Promise<unknown[]>`.

## Final state of the `pool` export (for reference)

After all five tasks the bottom of `src/pool.ts` should look like this:

```ts
export const pool = Object.assign(
	(concurrency = 10, options?: Omit<PoolOptions, 'concurrency'>): PromisePool =>
		new PromisePoolImpl({ ...options, concurrency }),
	{
		parallel: <T extends PromiseFunction[]>(
			commands: [...T],
			options?: PoolOptions,
		): Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }> => {
			if (!commands.length) return Promise.resolve([]) as Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>;
			const parallelPool = new PromisePoolImpl({ concurrency: Number.POSITIVE_INFINITY, ...options });
			for (const cmd of commands) parallelPool.enqueue(cmd);
			return parallelPool.close() as Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>;
		},
		serial: <T extends PromiseFunction[]>(
			commands: [...T],
			options?: Omit<PoolOptions, 'concurrency'>,
		): Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }> => {
			if (!commands.length) return Promise.resolve([]) as Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>;
			const parallelPool = new PromisePoolImpl({ ...options, concurrency: 1 });
			for (const cmd of commands) parallelPool.enqueue(cmd);
			return parallelPool.close() as Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>;
		},
	},
);
```

## Notes

- The `[...T]` spread in the parameter triggers tuple inference. Without it TypeScript collapses heterogeneous arrays to their union element type.
- `Awaited<ReturnType<T[K]>>` unwraps both `Promise<X>` and plain `X` return types, making the signature future-proof.
- The two `as` casts in each method are safe: the runtime values are correct; the casts bridge the gap between the untyped internal pool and the typed public return.
- `parallel` keeps `options?: PoolOptions` (allows overriding concurrency); `serial` keeps `options?: Omit<PoolOptions, 'concurrency'>` (concurrency is fixed at 1, so omitting it from the public API makes sense).
- These type changes are backwards compatible: existing callers that stored results in `any[]` or `unknown[]` continue to work unchanged.
